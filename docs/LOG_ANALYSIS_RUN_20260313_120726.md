## 1. GPS 对齐与 GPS 约束

### 1.1 配置与运行时矛盾（核心问题）

- **ConfigManager 在 load() 中**从 YAML 直接读并打印：
  - `[ConfigManager][GPS_DIAG] sensor.gps.enabled read from YAML = true`
  - `sensor.gps.topic = '/ublox/fix'`
- **AutoMapSystem 随后**打印：
  - `[AutoMapSystem][CONFIG] sensor.gps.enabled=false sensor.gps.topic=/ublox/fix`
- **LivoBridge**：
  - `[LivoBridge][GPS] init received gps_enabled=false`
  - `[LivoBridge][GPS] GPS disabled (sensor.gps.enabled=false from config). Trajectory CSV will have no GPS columns filled.`
  - `gps=disabled`

因此：**YAML 里是 true，但运行时用于 LivoBridge 的 gps_enabled 为 false**，导致未订阅 `/ublox/fix`，GPS 数据从未进入系统。

### 1.2 后果（与日志一致）

- 所有 keyframe：`has_gps=0`，BACKEND 行始终 `gps_win=0`。
- `[GPS_KF_BIND] found_gps=false window_size=0 gps_ts_range=[0.000, 0.000]` 反复出现。
- `[queryByTimestamp] result=not_found window_size=0`（GPS 窗口为空）。
- **无任何** `align_ok` / `try_align` / `GPS_ALIGN` / `add_constraint`（GPS 因子）相关成功日志。

结论：**GPS 对齐与 GPS 约束在本 run 中均未发生，且根因是“运行时认为 GPS 未启用”。**

### 1.3 潜在根因与建议

- `gps_enabled_from_config_` 来自 `ConfigManager::instance().gpsEnabled()`；而 load() 内用 `cfg_["sensor"]["gps"]["enabled"]` 直接读为 true。若 `gpsEnabled()` 走的是 `get<bool>("sensor.gps.enabled", false)` 且该路径因故返回默认值 false，就会出现上述矛盾。
- **建议**：
  1. 在 `ConfigManager::gpsEnabled()` 内打日志：当前走的是“直接 cfg_ 读”还是“get() 兜底”，以及最终返回值。
  2. 若存在多处 config 加载或覆盖，确保 **sensor.gps.enabled** 只从单一来源读一次并与 load() 内诊断一致。
  3. 在 LivoBridge 订阅前再次用 `ConfigManager::instance().gpsEnabled()` 做一次一致性检查并打 INFO，便于后续排查。

## 4. 其他问题与潜在风险

### 4.1 rosbag 话题忽略

- `rosbag2_player` 因缺少 `dvs_msgs`、`ublox_msgs` 等，忽略了例如 `/dvs/events`、`/ublox/navpvt`、`/ublox/navsat` 等。  
- `/ublox/fix` 若为 **sensor_msgs/NavSatFix**（标准类型），通常不会被 ublox_msgs 影响；但若实际 bag 里 fix 的 type 与 automap 订阅不一致，也可能收不到。需确认 bag 中 `/ublox/fix` 类型与 LivoBridge 订阅类型一致，且在 **GPS 启用后** 再次验证是否有 `LivoBridge][GPS]` 首条消息。

### 4.2 整段 run 的 loop 计数为 0

- 从首条 BACKEND 到日志末尾，**所有 BACKEND 行均为 `loop=0`**，与“无回环约束加入”一致。若设计上应有子图内/间回环，则本 run 未达成预期。

### 4.3 潜在配置读取 Bug

- 同上：**同一份 YAML 在 load() 里读到的 sensor.gps.enabled 为 true，而 gpsEnabled() 传给 LivoBridge 的为 false**。这属于潜在 bug，需在 ConfigManager 内统一来源（或显式缓存 load() 时的 enabled 值），并加诊断日志。

### 4.4 描述子格式与 TEASER

- `[INTRA_LOOP][PREPARE] ... (using fallback descriptor)` 表明当前使用 **fallback 描述子**，Overlap Transformer 未加载或未使用。若希望提升回环召回率，可检查模型路径与加载逻辑；TEASER 已启用（`LoopDetector] Initialized ... TEASER=enabled`），瓶颈更可能在“候选检索/相似度”或“约束写入”环节。
