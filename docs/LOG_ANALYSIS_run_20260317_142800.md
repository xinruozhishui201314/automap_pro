## 1. 已存在问题（结合日志与代码）

### 1.1 ingress_queue_max_size 配置与运行值不一致（严重）

**日志证据**：
- `CONFIG_FILE_PARAMS` / `CONFIG_SYSTEM_CACHE`: `ingress_queue_max_size=2000`
- `CONFIG_READ_BACK`: `system.ingress_queue_max_size=256`
- `AutoMapSystem][CONFIG]`: `ingress_queue_max_size=256`

**代码根因**：`automap_pro/include/automap_pro/core/config_manager.h` 第 61–71 行：

```cpp
size_t ingressQueueMaxSize() const {
    if (system_queue_cached_) {
        size_t v = static_cast<size_t>(system_ingress_queue_max_size_);
        if (v < 2) return 2;
        if (v > 256) return 256;   // ← 硬编码上限，YAML 写 2000 也被截断为 256
        return v;
    }
    ...
}
```

**影响**：配置文件中写 2000 无效，实际入口队列最大 256，高吞吐或短暂堆积时更容易触发背压/等待；与“CONFIG_SYSTEM_CACHE 与 CONFIG_READ_BACK 一致”的日志描述矛盾（缓存的是文件值 2000，read-back 是 getter 返回值 256）。

**建议**：  
- 若 256 为有意设计：在 YAML 注释或文档中说明“实际最大 256”，并让 CONFIG_READ_BACK 与 getter 一致（或去掉 CONFIG_SYSTEM_CACHE 中 ingress 的“与 CONFIG_READ_BACK 一致”表述）。  
- 若需支持 >256：去掉或提高该上限（如 2000），并做背压/内存测试。

### 1.3 体素下采样多次“overflow risk”重试

**日志证据**：全 log 共 **44 次**  
`[Utils][utils.cpp:161][voxelDownsample] voxelDownsample: overflow risk with leaf=0.2000, retrying with larger leaf`

**代码位置**：`automap_pro/src/core/utils.cpp` 约 156–162 行：

```cpp
for (int attempt = 0; attempt < 8; ++attempt) {
    if (!voxelGridWouldOverflow(...)) break;
    leaf *= 2.0f;
    if (attempt == 0) {
        ALOG_WARN("Utils", "voxelDownsample: overflow risk with leaf={:.4f}, retrying with larger leaf", leaf_size);
    }
}
```

**含义**：在 leaf=0.2m 下，点云 AABB 导致体素网格索引可能整数溢出，实现自动将 leaf 放大最多 8 倍重试；首次重试打 WARN。

**影响**：  
- 已做防护，不会崩溃，但**单帧下采样分辨率被临时放宽**（0.2→0.4 或更大），可能略微影响关键帧/匹配质量；  
- 若在关键路径上频繁触发，有轻微 CPU 与日志噪音。

**建议**：对“单帧小点云”可预先按 AABB 估算安全 leaf，或对 backend/keyframe 路径使用略大默认 leaf（如 0.25）减少重试；并监控 `VOXEL_OVERFLOW_DROPPED` 指标。

## 2. 潜在问题

### 2.1 子图未冻结前大量 GPS/关键帧 defer 与 pending 堆积

**日志现象**：  
- `addGPSFactor_defer sm_id=2 reason=node_not_exists`、`addGPSFactorForKeyFrame_defer ... reason=not_in_estimate` 反复出现（约 418 处 node_not_exists/defer 相关）；  
- `pending_values=54 pending_factors=55`、`current_estimate_ size=157`、`keyframe_count=209` 等：即大量关键帧节点与 GPS 因子尚未进入 ISAM2。

**代码逻辑**（与 [BACKEND_CONSTRAINT_ANALYSIS_20260317_FULL.md] 一致）：  
- 子图节点仅在 `onSubmapFrozen` 时加入（addSubMapNode）；  
- 当前活跃子图（如 sm_id=2）未冻结时，`node_exists_` 中无 s2，故子图级 GPS 正确 defer；  
- 关键帧节点已加入 pending，但要等 **forceUpdate**（子图冻结时触发）才 commit 进 ISAM2，因此 `current_estimate_` 远小于 keyframe_count。

**潜在影响**：  
- 子图 2 冻结前，几十个关键帧及对应 GPS 一直在 pending，**优化延迟**等于“当前子图持续时间”；  
- 若子图很大或冻结很晚，pending 会持续增长，单次 commit 规模变大，存在延迟与峰值计算量问题。

**建议**：可考虑“定时或按 pending 规模触发 forceUpdate”（在保证与 onSubmapFrozen 顺序一致的前提下），或监控 pending_factors/pending_values 与子图冻结间隔。

### 2.2 HBA 与 save/REBUILD 时序（若本 run 执行 finish_mapping）

文档 [HBA_GHOSTING_ROOT_CAUSE_20260317.md] 已说明：若 `hba_running_ = false` 在 `done_cbs_`（含 updateAllFromHBA + rebuildMergedCloud）之前执行，则 finish 线程会在 rebuild 完成前进入 save，导致重影。

本段日志未包含 `finish_mapping_save_enter` / `REBUILD_MERGE`，若同一次 run 的后续日志中有 finish_mapping，需确认：  
- `finish_mapping_save_enter` 是否出现在 `REBUILD_MERGE Done` 之后；  
- 若否，则存在与文档相同的 HBA 重影风险，应按该文档将 `hba_running_ = false` 移到所有 `done_cbs_` 之后。

### 2.3 ingress 硬上限 256 的背压与可观测性

见 1.1。配置 2000 实际 256，在高速率或后端偶发变慢时，入口队列更容易满，触发背压；若日志中出现 `backpressure`/`queue full` 等，可优先核对是否受此上限影响。建议在 CONFIG_READ_BACK 或启动日志中明确打印“生效的 ingress_queue_max_size=256（cap）”。

---

### 2.4 voxelDownsample 频繁 overflow 重试对质量与性能的长期影响

44 次重试说明 0.2m leaf 在本场景点云范围下经常不满足安全条件；虽然已自动放大 leaf，但：  
- 若这些帧参与关键帧或子图匹配，分辨率略降可能略微影响精度；  
- 重试循环有额外 CPU 与可能的延迟。建议对 backend/keyframe 路径统计“首次 leaf 即通过”的比例，必要时对高范围点云预选更大 leaf 或分块下采样（与现有 voxelDownsampleChunked 思路一致）。
