## 4. 严重问题：Backend 线程卡住

### 4.1 现象

- 约 **15:10:52** 起 backend 心跳超时：
  - `WARNING: slow threads: backend(14s)` → 19s → 24s → 29s。
  - 随后 **CRITICAL: threads stuck: backend(34s)**、**backend(39s)**。
- 同一时期：feeder 持续入队（如 `pushed frame seq=1400/1410/1420`），`frame_queue` 从 148 增至 199，**backend 不再 POP**，说明 backend 线程在某处阻塞或长时间计算未返回。

### 4.2 可能原因（需结合 gdb/栈与代码确认）

1. **ISAM2 优化**：回环或 GPS 因子加入后的 `update()` 耗时过长或死锁。
2. **HBA**：若该时段触发 HBA 或与 HBA 的交互，可能在 HBA/GTSAM 侧阻塞。
3. **锁竞争**：backend 与其它线程（如 loop_opt、map_pub）共享的锁长时间占用。
4. **GTSAM 单节点/首次 update**：与上述 GPS defer 逻辑相关的 `forceUpdate` 或首次 `commitAndUpdate` 路径异常。

建议：在出现 backend(14s) 的同一 run 上用 **GDB 或采样 profiler** 抓取 backend 线程栈，确认卡在 `IncrementalOptimizer::*`、`ISAM2::update` 还是 HBA/锁；并检查该时刻前后是否有大量回环/GPS 批量加入或 HBA 触发。
