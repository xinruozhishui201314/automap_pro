# Cursor 中文日常使用提示词模板

适用于大型项目的排障、重构、修改前审查、实施修改、回归检查与深度分析。

原则：
- 先分析，后修改
- 先缩小范围，再深挖
- 先看影响，再实现
- 区分事实、推断、猜测、缺失信息

---

## 1. 标准排障分析

请使用 `root-cause-investigator` agent 和 `root-cause-analysis` skill。

这是一个大型项目中的问题，请先不要修改代码。

背景信息：
- 现象：
- 预期：
- 触发条件：
- 最近变更：
- 当前已知相关模块：
- 本次允许修改范围：
- 本次禁止修改范围：

请按以下结构输出：
1. 问题现象总结
2. 已确认事实
3. 高概率推断
4. 当前缺失的信息
5. 候选根因（至少 3 个，按概率排序）
6. 每个根因的验证方法
7. 最小下一步排查动作

重点：
- 不要把报错点直接当成根因
- 优先分析调用链、数据流、状态流、异常传播
- 不要直接给修复代码

---

## 2. 只做根因分析，不进入实现

请只做根因分析，不要进入实现阶段。

使用 `root-cause-analysis` skill。

我希望你严格区分：
- 已确认事实
- 高概率推断
- 低置信度猜测
- 缺失证据

然后输出：
1. 最可能的 3 个根因
2. 每个根因各自解释了哪些现象
3. 每个根因有哪些现象还解释不完整
4. 最值得优先验证的一个点
5. 为什么这个验证点信息价值最高

不要修改代码。

---

## 3. 从报错点向上追根因

请使用 `root-cause-investigator` agent。

从当前报错点向上追溯，不要停留在异常抛出的地方。

请重点分析：
- 入口点
- 中间数据转换
- 状态变化
- 上下游依赖
- 哪一层最早出现了无效状态

输出格式：
1. 当前报错点只是症状还是根因
2. 上游可能的触发链路
3. 哪个环节最可能第一次引入错误状态
4. 应优先检查的文件 / 日志 / 配置
5. 还不能下结论的地方

---

## 4. 先缩小范围再深挖

这是一个大型项目问题，我不希望你直接展开很大的排查清单。

请先做“缩小范围”：
1. 先判断最可能是哪个层级的问题：
   - 入口参数
   - 应用服务
   - 领域逻辑
   - 数据访问
   - 外部依赖
   - 配置/环境
2. 给出你当前最怀疑的两个层级
3. 说明为什么优先怀疑它们
4. 给出最小的高信号排查动作

请不要直接写修复代码。

---

## 5. 排查偶发 bug

请使用 `root-cause-investigator` agent 和 `deep-analysis-sop` skill。

这是一个偶发问题，请重点考虑：
- 并发 / race condition
- 缓存不一致
- 异步处理
- 配置差异
- 历史数据异常
- 重试 / 超时 / 时序问题

背景信息：
- 现象：
- 出现频率：
- 是否和数据相关：
- 是否和环境相关：
- 最近是否发布过：
- 当前已有日志/trace：

请输出：
1. 偶发问题的主要候选原因
2. 各原因的触发条件
3. 如何验证每种可能性
4. 哪个验证动作性价比最高
5. 如果只有一次机会排查，最先看什么

---

## 6. 怀疑并发或缓存问题

请重点从以下方向分析，不要先改代码：
- 并发竞争
- 缓存击穿 / 缓存脏数据 / 缓存未失效
- 事务边界
- 异步任务顺序
- 重试导致的重复执行

请输出：
1. 哪些现象支持并发/缓存/事务问题
2. 哪些现象不支持
3. 代码静态分析能看出的风险点
4. 必须依赖运行态证据确认的点
5. 最小下一步验证动作

---

## 7. 性能问题先定位瓶颈

请使用 `root-cause-analysis` skill。

这是一个性能问题，请先不要给优化方案，先帮我定位瓶颈来源。

请从这些角度分析：
- 调用链耗时分布
- 数据访问
- 外部依赖
- 缓存命中
- 线程池 / 并发
- 序列化 / 反序列化
- 不必要的重复计算
- 配置差异

输出格式：
1. 从静态代码角度最可疑的瓶颈点
2. 哪些必须依赖运行态数据确认
3. 最有价值的指标 / 日志 / trace
4. 最小下一步验证动作
5. 在未确认瓶颈前，不要直接给优化 patch

---

## 8. 接口偶发超时分析

请用深度分析方式排查这个接口偶发超时问题。

不要直接假设是 SQL 或 Redis 问题。
请先建立候选原因树，至少包含：
- 数据库慢
- 外部调用抖动
- 锁竞争
- 线程池饱和
- 缓存异常
- GC / 配置问题
- 上游请求参数异常

然后输出：
1. 各候选原因的支持证据
2. 各候选原因还缺什么证据
3. 最小高信号排查路径
4. 最可能的前 3 个方向

---

## 9. 修改前先做影响范围分析

请使用 `impact-reviewer` agent 和 `impact-scope-review` skill。

现在不要修改代码，先做影响范围评估。

本次计划修改内容：
- 目标：
- 怀疑根因：
- 预计改动模块：
- 不允许改动模块：

请输出：
1. 可能受影响的模块 / 层 / 文件类型
2. 上下游契约风险
3. 数据 / 状态风险
4. 缓存 / 事务 / 异步 / 并发风险
5. 推荐补测范围
6. 哪些地方最容易漏改

---

## 10. 字段/接口变更前审查

我准备修改一个字段 / 接口定义。

请先使用 `impact-scope-review` skill，不要实现。

请重点检查：
- controller / service / repository
- DTO / VO / entity / mapper
- 前后端契约
- 序列化 / 反序列化
- 缓存 key 或缓存内容
- 测试、mock、fixture
- 文档和兼容性

输出：
1. 变更影响清单
2. 兼容性风险
3. 建议的最小改动范围
4. 必须同步更新的地方
5. 哪些地方容易造成隐性 bug

---

## 11. 先出改动计划，再实施

请使用 `fix-plan-generator` skill。

先不要直接改代码，先输出改动计划。

背景：
- 问题：
- 根因判断：
- 本次允许改动范围：
- 不允许改动范围：
- 特别约束（例如兼容性、发布时间、老模块不能动）：

请输出：
1. 修复目标
2. 改动边界
3. 计划修改的文件 / 模块
4. 每个文件为什么要改
5. 风险点
6. 为什么这是根因修复而不是表面补丁
7. 测试建议
8. 回滚 / fallback 建议

---

## 12. 开始实施，但要求控制范围

请使用 `code-change-executor` agent。

根因和改动计划已经确认。
现在可以开始实现，但请严格控制改动范围。

要求：
1. 修改前先列出计划改动的文件
2. 说明每个文件的修改目的
3. 只做最小必要改动
4. 不要顺手做无关重构
5. 保持兼容性，除非我明确允许破坏
6. 修改后输出变更总结、风险点和测试建议

---

## 13. 限定只能改某些目录

请实现这个修改，但必须遵守以下限制：

允许修改：
- app/
- domain/

禁止修改：
- legacy/
- shared-api/
- infra/

请使用 `code-change-executor` agent。

在动代码前先输出：
1. 计划修改文件
2. 为什么这些文件足够解决问题
3. 哪些地方你刻意不改
4. 当前方案的局限性

---

## 14. 修改后做系统性回归检查

请使用 `test-regression-reviewer` agent 和 `regression-checklist` skill。

针对这次修改，请检查：
- 已覆盖的正常路径
- 已覆盖的异常路径
- 可能遗漏的边界场景
- 并发 / 缓存 / 事务 / 异步风险
- 兼容性风险
- 回滚 / fallback 风险

请输出：
1. 已覆盖场景
2. 潜在遗漏场景
3. 推荐新增的单测 / 集成测试 / 回归测试
4. 上线前需要重点关注的风险
5. 是否建议灰度或加监控

---

## 15. 发布前做最终风险审查

请站在发布审查角度 review 这次修改。

重点检查：
- 是否存在未覆盖的边界情况
- 是否可能影响上下游契约
- 是否可能引入历史兼容问题
- 是否存在缓存 / 事务 / 异步 / 并发副作用
- 是否需要补日志、监控或告警

请按以下格式输出：
1. 发布阻塞项
2. 高风险项
3. 中风险项
4. 建议补充的测试
5. 建议补充的监控
6. 回滚触发条件

---

## 16. 先分析重构，不要直接重写

请使用 `refactor-architect` agent 和 `large-project-refactor-design` skill。

不要直接修改代码，先分析当前设计问题。

请从这些角度分析：
- 职责划分
- 模块边界
- 重复逻辑
- 数据模型一致性
- 异常传播
- 隐式耦合
- 历史兼容逻辑

输出：
1. 当前设计问题总结
2. 问题层级（函数 / 类 / 模块 / 架构）
3. 耦合点和约束
4. 三种方案：
   - 最小改动
   - 平衡演进
   - 理想目标
5. 每种方案的收益、风险、兼容性、成本
6. 推荐方案
7. 分阶段落地路径

---

## 17. 判断这是重构还是重写

请先判断这个任务到底属于：
- 局部清理
- 聚焦重构
- 跨模块重设计
- 伪装成重构的重写

请说明：
1. 为什么这样判断
2. 风险等级
3. 合理的实施边界
4. 如果坚持一次性大改，会有什么风险
5. 更可行的分阶段方案是什么

---

## 18. 要求给出三种重构方案对比

不要只给一个重构建议。

请至少给出三种方案：
1. 最小改动方案
2. 平衡方案
3. 理想方案

每个方案都必须说明：
- 改什么
- 不改什么
- 好处
- 风险
- 兼容性影响
- 迁移复杂度
- 是否适合分阶段上线

最后给出推荐，并说明为什么不是另外两个。

---

## 19. 限制范围的重构设计

这是一次受限重构。

允许修改：
- order-app/
- order-domain/

不允许修改：
- legacy/
- gateway/
- public-api/

请在这些限制下，设计最安全的重构路径。

输出：
1. 当前结构问题
2. 在限制下能解决的问题
3. 在限制下不能解决的问题
4. 推荐方案
5. 分阶段步骤
6. 会留下哪些技术债

## 20. 完整深度分析 SOP

请使用 `deep-analysis-sop` skill。

对这个问题执行完整的深度分析流程，不允许跳步，也不要直接进入实现。

请按以下阶段输出：
1. 问题定义
2. 范围与约束
3. 高信号上下文
4. 调用链 / 数据流 / 状态流 / 异常传播
5. 事实 vs 推断 vs 未知
6. 候选根因 / 候选设计问题
7. 验证方案
8. 影响范围
9. 解决方案选项
10. 回归与发布风险

如果信息不足，请明确指出最关键缺失信息，而不是泛泛地让我“补更多代码”。

---

## 21. 要求它挑战自己的结论

假设你当前的主要结论是错的，请反过来挑战自己。

请输出：
1. 当前主结论最薄弱的地方
2. 一个最强竞争解释
3. 哪些现象并不能完全被当前结论解释
4. 哪个证据最能区分这两个解释
5. 如果只能做一个验证动作，应该做什么

不要过早确定唯一答案。

## 22. 从架构视角分析问题

这个问题可能不是单点 bug，而是架构或设计问题。

请从以下角度分析：
- 数据流
- 状态流
- 模块边界
- 职责划分
- 异常传播
- 依赖方向
- 可观测性

输出：
1. 当前问题中暴露出的设计缺陷
2. 短期修复建议
3. 中期重构建议
4. 长期目标结构
5. 哪些改动不适合现在做

---

## 23. 识别系统性原因

不要只分析直接报错点。

请区分：
- 表面现象
- 直接原因
- 上游触发原因
- 系统性 / 设计层原因

1. 哪一层最像真正根因
