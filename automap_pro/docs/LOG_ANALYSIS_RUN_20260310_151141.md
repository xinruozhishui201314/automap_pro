# 日志分析：run_20260310_151141 后端异常

## Executive Summary

- **现象**：后端只处理了 1 帧（processed_no=1），之后一直处于 `waiting_for_data`，HEARTBEAT 显示 `frames_popped=4 processed_no=1 queue=0`。
- **根因**：**LivoBridge 在收到约 5 条 odom、4 条 cloud 后，不再收到新的 odom/cloud 回调**；fast_livo 进程持续发布到 #3000，说明数据在 bag→fast_livo 链路上正常，断点在前端订阅→automap_system 的 ROS2 回调交付。
- **最可能原因**：**Executor 所在线程在首条有效 GPS 回调前后被阻塞或不再执行回调**（单线程 Executor 下，若某回调长时间占用或存在死锁，后续 odom/cloud 无法被调度）。
- **已做**：强化 BACKEND/LivoBridge/FEEDER 日志与停滞诊断，便于下次复现时精确定位是「回调未执行」还是「ingress/queue 堵塞」。

## 2. 根因分析

### 2.1 数据流确认

- **fast_livo**：订阅 bag 的 lidar/imu，发布 `/aft_mapped_to_init`、`/cloud_registered`，日志显示已发布到 #3000 → 上游正常。
- **automap_system**：LivoBridge 订阅上述两个 topic，通过 `onOdometry`/`onCloud` 写 odom 缓存与 ingress → feeder → frame_queue → backend worker。
- 日志中 **FEEDER** 只出现 seq=1..4 的 push，**BACKEND POP** 只出现 frame_no=1..4，与 **LivoBridge 只收到 4 条 cloud** 一致；**odom #5 之后无 odom #6、无 cloud #5** → 断点在「ROS2 向 automap_system 投递回调」，而非 feeder/backend 逻辑错误。

### 2.2 为何回调停止交付？

在单线程 Executor 下，若某回调长时间不返回或发生死锁，后续消息不会被执行。时间顺序为：

1. cloud #4 回调返回后，odom #5 执行；
2. 随后首条有效 GPS 回调执行（ENU 原点、map_frame.cfg 写入、addGPSMeasurement）；
3. 日志显示 GPS 回调内的 MapFrameConfig::write 与 addGPSMeasurement 均完成（有 "map_frame.cfg written" 与 "First GPS measurement added"），**即该回调本身已返回**；
4. 此后本应继续执行 cloud #5、odom #6 等，但日志中未再出现任何 LivoBridge odom/cloud 日志。

因此更符合以下两种可能之一：

- **A. Executor 线程在 GPS 回调返回后不再被调度**  
  例如：进程内多节点/多 executor、或 launch 的 spin 方式导致执行 automap_system 回调的线程不再运行。
- **B. DDS/ROS2 层未再向该订阅投递**  
  例如：QoS 不匹配、分区、或已知的 best_effort 多订阅/背压导致后续消息未被投递。

当前日志无法区分 A/B，因此需要强化日志，在下次复现时直接看到：

- 是否仍有 `[LivoBridge][DATA] cloud #N` / odom 增长；
- HEARTBEAT 中的 `livo_cloud_count` / `livo_odom_count` 是否停滞；
- `ingress` / `frame_queue` 是否非空（若 LivoBridge 计数在涨而 queue 空，则问题在 feeder；若 LivoBridge 计数不涨，则问题在回调未执行或 DDS）。

## 3. 已采取的强化日志与诊断

- **BACKEND HEARTBEAT（waiting_for_data）**  
  - 增加：`livo_cloud_count`、`livo_odom_count`、`ingress_sz`、`frame_queue_sz`。  
  - 便于一眼看出：是「没有新数据」（LivoBridge 计数不涨）还是「有数据但卡在 ingress/queue」（计数涨、ingress/queue 非空）。

- **BACKEND_STALL_DIAG**  
  - 当「队列空且等待数据」超过约 60s，且 LivoBridge 的 cloud/odom 计数较上次无增长时，打一条 WARN，汇总：  
    `frames_popped`、`processed_no`、`livo_cloud_count`、`livo_odom_count`、`ingress_sz`、`queue_sz`，并提示：  
    - 若 LivoBridge 计数不再增加 → 检查 Executor/线程模型与 QoS；  
    - 若 ingress_sz>0 → feeder 可能卡住；  
    - 若 queue_sz>0 → worker 可能卡住。

- **LivoBridge**  
  - 对前 20 帧 cloud：在 `onCloud` 入口/出口打 INFO（ENTER/EXIT #N），用于确认 cloud #5 是否曾被调用。  
  - 对首条有效 GPS：在 `onGPS` 首条有效 fix 入口/出口打 INFO，确认首条 GPS 回调是否返回。

- **FEEDER HEARTBEAT**  
  - 在现有 HEARTBEAT 中增加当前 `ingress_queue_.size()`（在合法持锁处采样），便于确认数据是否堆积在 ingress。

以上日志均带固定 tag（如 `BACKEND HEARTBEAT`、`BACKEND_STALL_DIAG`、`LivoBridge`、`FEEDER`），便于 grep 快速定位。

## 4. 建议的后续排查（若再次复现）

1. **确认 Executor 与 spin 方式**  
   - 查看 launch 中 automap_system 的 `executor` 类型（SingleThreaded vs MultiThreaded）、以及 `spin` 是在主线程还是独立线程。  
   - 若为单线程，检查是否有其他回调或定时器长时间持锁/阻塞。

2. **临时关闭 GPS 或延迟写入 map_frame.cfg**  
   - 若关闭 GPS 后 odom/cloud 持续增长，可进一步缩小到「首条 GPS 相关路径」与 Executor 的交互（如写文件、锁、或回调组）。

3. **QoS 与 DDS**  
   - 确认 `/aft_mapped_to_init`、`/cloud_registered` 的发布端与 LivoBridge 订阅端 QoS 一致（depth、reliable/best_effort）；必要时用 `ros2 topic echo --qos-reliability reliable` 等对比。

4. **使用新日志**  
   - 再次运行后对日志执行：  
     `grep -E "BACKEND HEARTBEAT|BACKEND_STALL_DIAG|LivoBridge.*DATA|LivoBridge.*ENTER|LivoBridge.*EXIT|FEEDER.*HEARTBEAT"`  
   - 根据 `livo_cloud_count`/`livo_odom_count` 是否增长、`ingress_sz`/`queue_sz` 是否非空，判断断点是在「回调未执行」还是「ingress/queue 堵塞」。

## 5. 术语与缩写

| 术语 | 含义 |
|------|------|
| LivoBridge | 前端桥接：订阅 fast_livo 的 odom/cloud/kfinfo/gps，通过回调交给 AutoMapSystem |
| ingress | 从 LivoBridge 到 feeder 的队列，onCloud 只写 ingress，不直接写 frame_queue |
| frame_queue | feeder → backend worker 的队列，worker 按 process_every_n_frames 取帧做 KF |
| processed_no | 实际参与 tryCreateKeyFrame 的帧数；frames_popped 为从 frame_queue 弹出的帧数（含 skip） |
| HEARTBEAT | 后端/feeder 周期性打的存活与队列状态日志 |
